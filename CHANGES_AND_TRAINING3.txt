The critical problem I found
  
  The old data/npz_dataset.py loaded every shard fully into RAM (np.load + np.concatenate). I measured it: 2 local shards alone took the process
  from 210 MB → 1475 MB. Your full train set (part1 ~60 + part2 ~60 shards at ~700 MB each) would need tens of GB of RAM and instantly OOM on Kaggle
  (~30 GB). The smoke test only survived because it touched 2 shards. This had to be fixed before any GPU run.
  
  What I changed
  
  - data/npz_dataset.py (rewritten) — lazily memory-maps the uncompressed .npz shards. __init__ reads only the tiny .npy headers; each sample is
  paged off disk on demand. Verified: RAM stays flat (210 → 210 MB) and values match np.load exactly, including across shard boundaries. It also now
  accepts multiple roots and a flat shard dir, so it merges part1/train/ + part2/ (flat) for training while val/test stay on part1.
  - util/data_split.py — train_extra_roots (part2) is appended to the train split only.
  - config/base_config.py — added dataset.train_extra_roots, train.gpu, train.strategy.
  - config/ecrformer_kaggle_config.py (new) — Kaggle paths, 16-mixed precision (T4 has no bf16), DDP, 2 GPUs, train_bs=4 × accumulate=2 × 2 GPUs =
  effective batch 16, num_workers=2, checkpoints to /kaggle/working/experiments.
  - train.py — --devices/--strategy/--precision flags + DDP-aware trainer. Single-GPU behavior unchanged; smoke test still passes end-to-end.
  - kaggle_train.ipynb (new) — a validated 7-cell notebook that does everything.
  
  What you need to do on Kaggle
  
  1. Push this code somewhere Kaggle can read it. Easiest: zip the FDCMformer folder and upload it as a new Kaggle Dataset (e.g. fdcmformer-code).
  The notebook auto-detects it by finding train.py.
  2. Create a new Notebook, and in the right sidebar:
    - Accelerator → GPU T4 ×2
    - Internet → ON (required — LPIPS/VGG weights download on first validation)
    - Add data: attach all three — your code dataset, sen12ms-cr-spring-part1, sen12ms-cr-spring-part2.
  
  3. Import kaggle_train.ipynb (File → Import Notebook) or paste its cells. Run cells 1–3 first to confirm the shard counts (you should see ~60
  train in part1, 4 val, 19 test, ~60 in part2).
  4. Save Version → Save & Run All (Commit) to train headless on both T4s.
  
  Things to know
  
  - 9-hour limit: training auto-resumes from last.ckpt. To continue past a timeout, attach the previous run's output as an input and copy
  experiments/ into /kaggle/working/ (cell 5 + the resume markdown explain this).
  - Effective batch = 16, matching the original single-GPU recipe.
  - If you ever see NaNs from mixed precision, rerun cell 4 with --precision 32. If GPU OOM, lower train_bs and raise accumulate_grad_batches.
  
  One thing I could not verify here: an actual 2×T4 DDP run and the LPIPS download, since this is a Windows CPU box with no shards beyond the 2
  local ones. The data-loading, config, multi-root merge, and pipeline wiring are all verified; the DDP/precision path will first exercise on
  Kaggle.


  What I added for a professional 50–80 epoch run
  
  Training schedule (the key correctness fix): the LR scheduler milestones were hardcoded [120,150,170,180,190,200] for a 200-epoch run — at 80
  epochs the learning rate would have never decayed. I made the schedule config-driven (lr_milestones, lr_gamma) and set the Kaggle config to
  max_epoch=80, milestones [40,60,70,75], early_stop=15.
  
  Robust resume:
  
  - Fixed a real resume-breaking bug: the model pickled the whole config into the checkpoint, which fails to load under torch ≥2.6's
  weights_only=True. Now ignored (save_hyperparameters(ignore=['config'])); verified a full save→resume cycle restores all states in-place.
  - Rolling last.ckpt refreshed every 30 min and every epoch, so a mid-epoch kill loses little.
  - Notebook now auto-copies a previous run's experiments/ from /kaggle/input so resume is one click across the 9h limit.
  
  Graceful stop: max_time='00:08:30:00' — Lightning stops cleanly and checkpoints ~30 min before Kaggle's hard 9h kill (no lost progress).
  
  Observability / hygiene: keeps top-3 best checkpoints named best-{epoch}-{valid_loss}.ckpt, a LearningRateMonitor, log_every_n_steps=20, and a
  human-readable config_snapshot.json written next to the logs each run.
  
  All configurable from config/ecrformer_kaggle_config.py (epochs, schedule, save_top_k, ckpt_every_n_min, max_time); base defaults preserve the
  original ECRformer behavior.
